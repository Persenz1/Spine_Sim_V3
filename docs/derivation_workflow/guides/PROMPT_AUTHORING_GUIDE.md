# 机理推导提示词编写指导

> 用途：指导 Codex 为 A1–C3、大模块集成和全局集成分别编写任务提示词。
>
> 本指导不提供统一提示词套话。每份提示词应直接、具体地说明当前任务要做什么，但必须覆盖本文件规定的阅读输入和内容要求。

## 1. 编写提示词前的通用必读内容

Codex 必须完整阅读：

1. 最新正式 `engineering_fixed_context/engineering_fixed_context.md`；
2. `docs/extract/MECHANISM_MODULE_PLAN.md`；
3. `docs/extract/LITERATURE_MODULE_ROUTING.md`；
4. `docs/derivation_workflow/DERIVATION_WORKFLOW.md`；
5. 本指导；
6. 当前任务会使用的所有输出合同或模板。

不得仅凭文件标题、目录名、摘要或先前记忆编写正式提示词。

## 2. 按任务类型追加的必读内容

### 2.1 A1

- 无既有 `A_MODULE_CONTEXT`；
- 阅读 A1 的目标、边界、完成判据和最终最小文献包；
- 核对工程事实中与表面、针尖、碰撞、拖拽和数值事件有关的全部条目。

### 2.2 A2/A3、B2/B3、C2/C3

- 完整阅读当前大模块最新 `MODULE_CONTEXT`；
- 明确本轮必须保留的既有内容和允许修改的内容；
- 阅读当前子模块最小文献包的路由作用与使用边界。

### 2.3 B1 和 C1

- B1 完整阅读 `A_TO_B_CONTRACT.md`；
- C1 完整阅读 `B_TO_C_CONTRACT.md`；
- 只在明确审计需要时追加上游完整集成模型。

### 2.4 大模块集成

- 最新正式工程事实；
- 当前大模块包含三个子模块的最新完整 `MODULE_CONTEXT`；
- `MECHANISM_MODULE_PLAN` 中该大模块的完成判据和上下游合同要求；
- 默认不重新上传论文和历史快照，除非发现明确证据缺口或冲突。

### 2.5 全局集成

- 最新正式工程事实；
- `A_INTEGRATED_MODEL.md`、`B_INTEGRATED_MODEL.md`、`C_INTEGRATED_MODEL.md`；
- A→B、B→C 合同的嵌入章节；
- 全局一致性检查要求。

## 3. 每份提示词必须显式包含的内容

### 3.1 任务身份

- 模块/阶段编号；
- 提示词版本；
- 输入文件版本或基线；
- 本轮唯一目标。

### 3.2 研究背景与系统位置

提示词必须先用足够具体的自然语言说明：

- 研究的机器人、机构或物理对象是什么；
- 面向哪些材料、工况和实验场景；
- 全系统从低层到高层的物理依赖链；
- 当前模块位于依赖链的哪一层；
- 当前结果为什么是后续模块的必要输入；
- 与本轮直接相关的固定几何、运动和加载背景。

不得让 Pro 仅凭“A1”“表面生成”等内部编号猜测研究背景，也不得把项目任务写成脱离钩爪式爬壁机器人爪刺啮合问题的一般数学题。

### 3.3 输入文件清单与阅读要求

逐项列出实际上传文件，并说明：

- 仓库相对路径或网页端可识别的准确文件名；
- 文件用途；
- 权威优先级；
- 是否必须完整阅读；
- 文件之间发生冲突时如何处理。

提示词本身必须包含该清单；`INPUT_MANIFEST.yaml` 只做本地复现记录，不能代替提示词中的说明。

对子模块执行任务，上传清单通常必须显式包含：最新工程事实、`MECHANISM_MODULE_PLAN.md`、当前提示词、当前输出所适用的全部模板、当前模块上下文或上游合同（如适用）以及逐个列名的最小文献包。只在 Codex 本地使用的工作流、全量文献路由和运行产物处理指导不默认上传，但其相关约束必须已经写入当前提示词。

### 3.4 范围与禁止越界

- 本轮必须解决的问题；
- 明确不处理的内容；
- 留给后续阶段的内容；
- 禁止重新定义的上游或低层机理；
- 首版明确排除的复杂度。

### 3.5 必答问题和完成判据

- 逐项列出 `MECHANISM_MODULE_PLAN` 对当前阶段要求回答的问题；
- 将抽象目标转换成可检查的理论结果、接口、算法或验证项；
- 明确什么情况下本轮不算完成。

### 3.6 证据使用策略

- 列出最小文献包及每包不可替代作用；
- 写明每篇文献不能被外推的边界；
- 备用文献不默认使用；
- 文献不足时允许使用 GPT 自身通用知识、互联网公开资料和公开数据库；
- 外部数值必须说明对象、尺度、单位、版本、迁移条件和网址；
- 无材料专属证据时采用参数化、候选范围或待标定项，不得硬编码唯一值。

### 3.7 输出要求

对子模块任务，至少要求：

1. 最新完整 `MODULE_CONTEXT`；
2. 最新完整工程事实候选；
3. `RUN_UPDATE_SUMMARY.yaml`；
4. 仅本地归档的 `CITATION_BRIEF.md`。

输出只受相应内容合同约束，不要求统一写作风格。大模块集成和全局集成只需完整、一致、可执行的正式机理推导；A/B 集成另须给出下游合同。

### 3.8 工程事实变更规则

- 不得静默修改正式工程事实；
- 机理方程和算法不得写入工程事实候选；
- 外部资料不能自动把候选提升为正式事实；
- 若无变化，工程事实候选完整复用正式基线；
- 任何候选变化必须在运行摘要中给出理由和结构化映射。

### 3.9 自检要求

提示词必须要求 Pro 在输出前检查：

- 是否完整回答所有必答问题；
- 是否遵守工程事实、层级边界和范围排除；
- 符号、单位、力和功方向是否一致；
- 状态、事件、输入和输出是否闭合；
- 是否存在未经标注的硬编码、伪造来源或过度外推；
- 是否保留未决问题和后续交接。

## 4. 提示词写作原则

- 直接说明任务，不写无作用的角色扮演或泛化口号；
- 内容要求优先于排版要求；
- 同一要求只在最合适的位置出现一次；
- 使用明确的“必须、不得、仅当”表达关键约束；
- 不要求 Pro 生成仓库代码，除非当前任务明确是实现任务；
- 不把本地机械归档、YAML 排障或文件修复交给人工。

## 5. Codex 定稿检查

在交付提示词前，Codex 必须自行确认：

- 所有必读文件已经实际完整阅读；
- 提示词已明确说明研究背景、系统对象和当前模块位置；
- 输入清单与准备上传的文件一致；
- 文献编号和最小包与最新路由台账一致；
- 输出要求引用了最新合同或模板；
- 没有要求人工参与 YAML、diff、路径、命名或生成器校验；
- 提示词版本、路径和工程事实版本正确。
